# Document Project Structure

## Overview

Use this skill to update repository documentation from the actual filesystem, not from assumptions or examples. Prefer concise trees that show the important project boundaries, entry points, configuration files, tests, and documentation assets.

## Workflow

1. Inspect the repository before editing.
   - Run `rg --files` to list tracked-looking project files quickly.
   - Run `find . -maxdepth 4 -type d | sort` when directory hierarchy matters.
   - Read the current `README.md` before choosing where to insert or update content.

2. Identify what belongs in the public structure.
   - Include source directories, tests, docs, project configuration, and workflow files.
   - Exclude `.git`, caches, virtual environments, generated bytecode, coverage outputs, and local editor files.
   - Do not document files that do not exist just because they appeared in an example prompt.

3. Write the tree in a stable, readable format.
   - Use a fenced `text` code block.
   - Use folder markers consistently when the README already uses them.
   - Keep comments short and useful: describe responsibility, not implementation trivia.
   - Collapse deep or repetitive folders when listing every file would obscure the architecture.

4. Place the section where it helps the reader.
   - Prefer an existing architecture, project structure, or repository layout section.
   - If none exists, create a short section near the architecture or quickstart content.
   - Preserve the README's existing language, tone, heading style, and formatting conventions.

5. Verify the edit.
   - Re-read the edited README section.
   - Run `git diff -- README.md` or the relevant documentation file.
   - Mention whether changes are documentation-only and whether tests were skipped.

## Tree Style

Use this shape when the repository does not already impose a different style:

```text
├── 📁 docs/
│   └── assets/readme/              # imagens e recursos usados no README
├── 📁 src/
│   ├── core/                       # fluxos principais da aplicação
│   └── utils/                      # utilitários, integrações e serviços compartilhados
├── 📁 test/                        # testes unitários, integração e suporte local
├── Makefile                        # comandos padronizados de desenvolvimento
├── pyproject.toml                  # dependências e configuração Python
└── README.md                       # documentação principal do projeto
```

## Documentation Rules

- Prefer the repository's existing language. In Portuguese READMEs, keep section text and comments in Portuguese.
- Avoid exhaustive trees for large codebases; show representative structure and summarize nested modules.
- Keep file comments factual and current. If the purpose is unclear, inspect the file instead of guessing.
- Preserve user edits and unrelated README content.
- Do not add extra skill resources unless the workflow becomes repetitive enough to justify scripts or references.
